## 一、整体训练架构

### 1.1 任务概述

四足机器人VBot在Section01赛道上完成全程导航：

```
START → 笑脸收集 → 红包收集 → 2026平台(庆祝) → 上楼梯 → 吊桥 → 下楼梯
      → 丙午大吉平台(庆祝) → 河床红包 → 滚球区 → 终点(庆祝)
```

- **全程距离**：Y轴 -2.4m → 32.0m，总长约 34.4m
- **导航路点**：24个硬编码waypoints（索引 0~23）
- **里程碑平台**：3个（wp6=2026平台, wp14=丙午大吉, wp23=终点）
- **地形类型**：坑洼、楼梯上行/下行、吊桥、河床、滚球区、不规则地形

### 1.2 观测与动作空间

| 维度 | 组成 |
|------|------|
| **观测 67维** | 线速度(3) + 陀螺仪(3) + 重力投影(3) + 关节角度(12) + 关节速度(12) + 上步动作(12) + 速度命令(3) + 位置误差(2) + 朝向误差(1) + 距离(1) + 到达标志(1) + 停稳标志(1) + 接触力(13) |
| **动作 12维** | 4条腿 × 3关节（髋/大腿/小腿） |

## 二、多点Checkpoint导航系统

### 2.1 24-Waypoint路径规划

采用**硬编码waypoint序列**，而非让RL自由探索路径：

```
阶段1 (wp0~6):  笑脸×3 + 红包×3 + 2026平台
阶段2 (wp7~14): 楼梯口 → 上楼梯×4级 → 吊桥红包 → 下楼梯 → 丙午大吉
阶段3 (wp15~23): 河床红包×6 → 滚球区 → 终点
```

**设计理由**：  
- 赛道路径固定（不是随机迷宫），硬编码路径消除了路径规划的不确定性
- RL策略只需专注学习**运动控制**（怎么走），不用学**路径规划**（往哪走）
- 路点间距约1~3m，给策略提供持续且密集的引导信号

### 2.2 Waypoint递进奖励

```python
到达奖励 = 5.0 + wp_idx × 0.5
```

| waypoint | 到达奖励 | 累积意义 |
|----------|----------|----------|
| wp0 | 5.0 | 起步 |
| wp6 (2026平台) | 8.0 | 第一阶段完成 |
| wp14 (丙午大吉) | 12.0 | 楼梯通过 |
| wp23 (终点) | 16.5 | 全程完成 |

**设计理由**：越远的waypoint奖励越大，形成递增梯度，防止策略学会"在前几个waypoint反复刷分"。

### 2.3 里程碑Gating机制

里程碑waypoint（wp6, wp14, wp23）**强制阻止前进**，必须完成庆祝动作后才放行：

```python
block_advance |= (wp_idx == milestone_idx) & (~celebration_done)
```

**设计理由**：比赛规则要求在平台执行庆祝动作，gating确保策略不能跳过。

## 三、课程学习（Curriculum Learning）

### 3.1 空间课程

```python
50%环境：从起点出发（学习完整路线）
50%环境：随机散布在赛道Y=[−2.4, 37.0]范围（学习各段地形）
课程出生高度：2.0m（确保在楼梯/吊桥/河床上安全着陆）
```

**设计理由**：  
- 50%起点环境保证策略学习完整序贯决策
- 50%随机环境让策略接触到**后半程地形**（楼梯、吊桥），不必等前半程收敛
- 2.0m空中出生避免卡在地形凹槽中

### 3.2 朝向随机化

```python
初始朝向：+Y方向 ± 20°随机
```

**设计理由**：比赛中机器人朝向可能有微小偏差，±20°训练增强鲁棒性。

## 四、地形自适应奖励系统

### 4.1 地形区域划分

根据Y坐标自动判断地形类型并调整奖励参数：

| Y范围 | 地形类型 | 目标速度 | 前进奖励倍数 |
|-------|----------|----------|-------------|
| Y < 8 | 坑洼/平地 | 1.0 m/s | 1.2× |
| 8 < Y ≤ 12 | 过渡区 | 0.8 m/s | 1.2× |
| 12 < Y ≤ 18 | **上楼梯** | **0.6 m/s** | **1.8×** |
| 18 < Y ≤ 21.5 | **下楼梯/吊桥** | **0.4 m/s** | **0.8×** |
| Y > 21.5 | 河床/滚球/终点 | 1.0 m/s | 1.2× |

**设计理由**：  
- 上楼梯需要慢速精准步态 → 低目标速度 + 高前进倍数（鼓励缓慢但坚定地上楼）
- 下楼梯冲太快会翻倒 → 最低目标速度 + 低前进倍数（抑制冲锋）
- 训练结果验证：25000步时下楼梯速度过快是主要失败原因，地形自适应直接针对此问题

### 4.2 下楼梯专项惩罚

```python
# 速度惩罚：下楼梯超过0.6m/s就罚
descent_speed_penalty = -0.05 × (speed - 0.6)

# 倾斜惩罚：下楼梯倾斜>45°就重罚
descent_tilt_penalty = -0.15 × (0.7 - tilt_cos)
```

**设计理由**：训练数据显示下楼梯段是失败率最高的区域，需要专项约束。

### 4.3 楼梯通用稳定性惩罚

```python
# 所有楼梯段倾斜>50°重罚
stair_stability_penalty = -0.1 × (0.6 - tilt_cos)  [when tilt>50°]
```

## 五、侧翻与倾斜检测体系

### 5.1 终止条件：IMU Roll/Pitch检测

```python
roll, pitch = _compute_roll_pitch_from_quat(root_quat)  # 四元数→欧拉角
终止条件：|Roll| > 60° 或 |Pitch| > 60°
终止惩罚：-10.0
```

**对比旧方案**：  
- 旧方案：`gravity_z > cos(70°)` — 只能检测"总倾斜角"，无法区分侧翻(Roll)和前后仰(Pitch)
- 新方案：独立检测Roll和Pitch，更精确地捕捉不同方向的失稳

**阈值选择依据**：  
- 60°而非70°：提前15%的角度终止，让策略学会在接近危险前就调整姿态
- 坑洼地形30-45°倾斜正常，60°已经是真正的侧翻

### 5.2 软倾斜惩罚（渐进式）

```
0° ~ 45°:  无惩罚（坑洼地形正常倾斜）
45° ~ 60°: 渐进惩罚 -0.002 × (0.707 - tilt_cos) / 0.707
60° 以上:  终止 + 强惩罚 -10.0
```

**设计理由**：三层递进防护 — 软警告 → 强惩罚 → 直接终止。

### 5.3 其他终止条件

| 条件 | 阈值 | 惩罚 |
|------|------|------|
| Y方向掉落悬崖 | Y < -3.5 | -30.0 |
| 超出地图前方 | Y > 35.0 | 终止 |
| 物理发散 | NaN检测 | 终止 |
| 卡住不动 | 连续400步速度<0.05 且未到达终点 | 终止 |
| 后退接近悬崖 | Y接近spawn-1m | -0.5/步 (软惩罚) |

## 六、庆祝动作系统

### 6.1 动作覆盖（Action Override）

**核心创新**：不依赖RL策略发现12维目标姿势（高维空间极难收敛），而是在庆祝阶段**直接PD控制绕过策略网络**。

```
RL策略职责：走到平台 → 减速停下（简单的导航+减速任务）
系统接管：检测到庆祝阶段 → PD控制执行预设姿势（100%成功率）
```

### 6.2 庆祝流程

```
阶段1 - Hold (1.5秒)：PD控制到默认站姿，平稳停下
阶段2 - Pose (1.5秒)：PD控制到庆祝预设姿势
```

**预设姿势（"抬前腿蹲坐"）**：
```python
FR: hip=0.0, thigh=-0.3, calf=-0.5   # 前腿抬起
FL: hip=0.0, thigh=-0.3, calf=-0.5   # 前腿抬起
RR: hip=0.0, thigh=1.5,  calf=-2.5   # 后腿下蹲
RL: hip=0.0, thigh=1.5,  calf=-2.5   # 后腿下蹲
```

### 6.3 庆祝奖励

| 奖励项 | 数值 | 说明 |
|--------|------|------|
| 到达里程碑 | +50.0 | 一次性 |
| 平台驻留 | +0.02/步 | 在中心1m内 |
| 静止驻留 | +0.03/步 | 速度<0.18 |
| 姿势接近 | ≤+0.5/步 | exp(-误差×2)×0.5 |
| 庆祝完成 | **+25.0** | 姿势保持1.5秒 |

**总庆祝回报**：约 50 + 25 + ~75步×0.5 ≈ **112.5分/平台**，远高于跳过平台继续前进的收益。

## 七、PPO超参数优化

### 7.1 当前超参数 vs 默认值

| 参数 | 默认/基线 | **当前值** | 调整理由 |
|------|----------|-----------|---------|
| `num_envs` | 2048 | **4096** | 甜区：课程学习覆盖好 + CPU压力可控。8192实测更慢（Python循环瓶颈已修复但内存压力大） |
| `learning_rate` | 3e-4 | **1.5e-4** | 多地形任务更复杂，降低学习率防止震荡 |
| `discount_factor` | 0.99 | **0.995** | 全程34m需要长视野，0.995让策略更重视远期waypoint回报 |
| `lambda_param` | 0.95 | **0.97** | 配合高γ，GAE更信任远期回报估计 |
| `entropy_loss_scale` | 0.0 | **0.008** | 适度探索，防止在楼梯/桥段陷入局部最优 |
| `rollouts` | 24~48 | **32** | 平衡：足够覆盖一个episode的多步决策，不过长导致数据过时 |
| `mini_batches` | 3~32 | **64** | batch=4096×32=131072样本，mini_batch=2048，梯度更稳定 |
| `share_policy_value` | True | **False** | 分离设计：policy关注动作精度，value关注全局地形/进度评估 |

### 7.2 网络架构

| 网络 | 架构 | 设计理由 |
|------|------|---------|
| **Policy** | (256, 256, 128) | 等宽前两层避免信息瓶颈，67→12维映射不需要太宽 |
| **Value** | (512, 256, 128) | 首层更宽：价值网络需要理解67维观测中的全局状态（位置、地形、进度） |

**为什么分离？**  
- Policy目标：输出精准的12维关节动作
- Value目标：评估"当前状态距离完成全程还有多少价值"
- 两者所需的特征提取完全不同，共享特征会互相干扰

### 7.3 超参数优化依据

| 优化项 | 问题现象 | 解决方案 | 依据 |
|--------|---------|---------|------|
| γ=0.99→0.995 | 策略只学会走到wp6就停滞 | 提高γ让远期waypoint(wp14+)的回报信号传回 | PPO理论：γ决定"回报视野"，0.99约100步，0.995约200步 |
| lr=3e-4→1.5e-4 | 奖励波动大，不稳定 | 降低学习率 | 多任务(导航+爬楼+庆祝)梯度方向冲突 |
| entropy=0.008 | 楼梯段策略卡死不转向 | 保留适度随机探索 | 0.01太随机，0.005太确定，0.008经验折中 |
| separate P/V | 学习平台期长 | 分离特征提取 | locomotion任务的标准做法（legged_gym默认False） |
| mini_batches=64 | 梯度噪声大 | 增大mini_batch | 4096envs×32rollouts=131K样本，64个batch每个2K样本 |

| 新区域探索 | +1.0 | 一次性 | 突破最远距离 |
| **Waypoint递进** | 5.0 + idx×0.5 | 一次性 | 到达新路点 |

| 下楼梯超速 | -0.05×超速 | 持续 | 下楼梯>0.6m/s |

| 翻倒终止 | -10.0 | 终止 | Roll/Pitch>60° |
| 悬崖掉落 | -30.0 | 终止 | Y<-3.5 |
| 旋转探索 | ≤0.6/步 | 条件 | 卡住>80步时鼓励转向 |
| 后退惩罚 | -0.5/步 | 持续 | 接近后方悬崖 |

### 8.2 奖励设计原则

1. **密集信号 > 稀疏信号**：速度跟踪+前进投影提供每步信号，waypoint奖励提供中期目标
2. **递增激励**：越远的waypoint奖励越大，防止前段刷分
3. **惩罚层次化**：软警告(45°) → 中惩罚(50°楼梯) → 硬终止(60°)，给策略调整空间
4. **地形感知**：不同地形不同速度目标，避免"一刀切"导致楼梯失败

## 九、性能优化

### 9.1 向量化

将关键热路径中的Python `for env_idx in range(num_envs)` 循环替换为NumPy批量操作：

- `_update_target_marker()`: 向量化DOF赋值
- `_update_heading_arrows()`: 批量四元数计算
- `_euler_to_quat_batch()`: 新增批量欧拉角→四元数方法
- reset中的四元数循环: 使用batch方法

**效果**：消除了num_envs增大时的线性Python开销，使4096环境成为可能。

### 9.2 动作滤波

```python
filtered = 0.6 × new_action + 0.4 × prev_filtered
```

**作用**：平滑策略输出，减少关节抖动，提高步态稳定性。
